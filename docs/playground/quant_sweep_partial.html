🎯 A fő finding

Az int4 "+1.4pp win" NEM univerzális — hanem méret-függő regularizáció. Kis hálón (nf=32) nagyobb a win (+2.0pp), közepesen (nf=64) még nyer (+1.4pp), de nagyobb hálón (nf=96) már veszít (−1.6pp). Az eredeti állítás overclaim volt.

💡 A revideált állítás

Eredeti overclaim (nf=64, egy adatpont): "Az INQ int4 BEVERI a float32-t (+1.4pp)."

Pontosabb állítás (3 adatpont): Az INQ int4 a méret függvényében:

  • Kis háló (nf=32): +2.00pp — regularizáció nyer
  • Közepes (nf=64): +1.40pp — még nyer
  • Nagy (nf=96): −1.60pp — már veszít

Gyakorlati következmény: a VRAXION célmérete (kis, mobilra/IoT-ra tervezett modell) PONT AZ A ZÓNA ahol az INQ int4 nyer. Nagyobb hálón nem kell, és nem is segít — csak a memória 8×-os megtakarítását adja −1.6pp áron.

Hátralévő: Code nf=32 int4, Code nf=32 binary, Code nf=96 ×3. ~25 perc.
