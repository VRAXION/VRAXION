# D9.0d Progressive Planet Renderer

Snapshot-mode HTML renderer for the D9.0d direct-genome landscape.
Pure HTML/Canvas, no external dependencies, no HTTP server required.

## Files in this directory

| File | Purpose |
|---|---|
| `progressive_planet.html` | The renderer. Open by double-click. Loads `state.js` from the same dir. |
| `state_mock.js` | Demo state fixture. Rename/copy to `state.js` to preview without a real run. |
| `README.md` | This file. |

## How to use

### Quick demo (no Python needed)

```
cd tools/d9_0d_progressive_planet
cp state_mock.js state.js          # or copy by hand on Windows
# double-click progressive_planet.html
```

The HTML is a snapshot renderer:
- It loads `state.js` once on page load via a `<script>` tag.
- The `Reload data` button (or F5) does a cache-busted reload.
- An optional `auto-reload 60s` checkbox is provided (default OFF).

There is NO live polling and NO HTTP server requirement. `file://` works.

### Real run (after the sampler exists)

The Python tools (built separately) generate a real `state.js` into a
dated output directory. Convention:

```
output/phase_d9_0d_progressive_planet_<YYYYMMDD>/
    progressive_planet.html       <-- copied from this template
    state.js                      <-- generated by the sampler
    progressive_atlas_history.jsonl
```

The build script must:
1. Copy `tools/d9_0d_progressive_planet/progressive_planet.html`
   into the run directory.
2. Write `state.js` next to it as
   `window.ATLAS_STATE = {...};`

## Schema

`state.js` must define `window.ATLAS_STATE` with `schema_version: "d9.0d-1"`.

Required top-level fields:
- `schema_version`, `run_id`, `last_updated`, `phase_status`
- `config` (H, lat_bins, lon_bins, eval lengths, checkpoint)
- `tiles[]` (see below)
- `queue[]`
- `progress`

Required per-tile fields:
- `tile_id`, `lat_bin`, `lon_bin`, `x`, `y`, `z`
- `n_scout`, `n_confirmed`, `target_n`
- `mean_delta`, `best_delta`, `median_delta`, `std_delta`
- `cliff_rate`, `positive_rate`, `mean_behavior_distance`, `confidence`
- `state` (enum, see below)
- `recommended_action`
- `per_type` map with keys exactly: `edge`, `threshold`, `channel`, `polarity`
- `scan_priority`, `split_priority`

Tile state enum (exact strings):
```
UNKNOWN, SCOUT, DESERT, CLIFFY, NOISY, PROMISING,
CONFIRMED_GOOD, RETIRED, SPLIT_CANDIDATE
```

Per-type entry fields (use `null` for missing, never omit the key):
```
n, mean_delta, best_delta, std_delta, cliff_rate, positive_rate
```

The renderer validates the schema on load. If `schema_version` does not match
or required fields are missing, the renderer shows a visible error instead of
silently rendering wrong values.

## Controls

| Control | Effect |
|---|---|
| Drag | Rotate the planet |
| Wheel | Zoom |
| Metric buttons | Change the height/color metric |
| Mutation type buttons | Filter tiles to a single mutation kind |
| Eval layer buttons | (visual hint, sampler-side filter coming) |
| Reload data | Force fresh load of state.js |
| auto-reload 60s | Periodic forced reload |

## Why type-toggle is mandatory

The D9.0b audit found a TYPE_SPLIT verdict:

| type | low-radius cliff rate | classification |
|---|---:|---|
| edge | 0.13 | rugged/searchable |
| threshold | 0.30 | rugged/searchable |
| channel | 0.42 | cliffy |
| polarity | 0.82 | cliffy |

Aggregating these into one map would hide the signal. Per-type filtering is
the load-bearing UI feature.
